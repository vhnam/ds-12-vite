# Using Vite+, the Unified Toolchain for the Web

This project is using Vite+, a unified toolchain built on top of Vite, Rolldown, Vitest, tsdown, Oxlint, Oxfmt, and Vite Task. Vite+ wraps runtime management, package management, and frontend tooling in a single global CLI called `vp`. Vite+ is distinct from Vite, and it invokes Vite through `vp dev` and `vp build`. Run `vp help` to print a list of commands and `vp <command> --help` for information about a specific command.

Docs are local at `node_modules/vite-plus/docs` or online at https://viteplus.dev/guide/.

## Review Checklist

- [ ] Run `vp install` after pulling remote changes and before getting started.
- [ ] Run `vp check` and `vp test` to format, lint, type check and test changes.
- [ ] Check if there are `vite.config.ts` tasks or `package.json` scripts necessary for validation, run via `vp run <script>`.
- [ ] If setup, runtime, or package-manager behavior looks wrong, run `vp env doctor` and include its output when asking for help.

## Design token pipeline

```
Tokens Studio export
  → packages/design-tokens/src/assets/tokens.json          [DO NOT EDIT in repo]
  → vp run @ds-12/design-tokens#build:tokens (Style Dictionary)
  → packages/design-tokens/src/tokens.generated.css        [AUTO-GENERATED — do not hand-edit]
  + packages/design-tokens/src/tokens.web.css              [HAND-EDITED — web extensions]
  → vp run @ds-12/design-tokens#generate:theme
  → packages/design-tokens/src/tokens.theme.css            [AUTO-GENERATED — do not hand-edit]
  → packages/ui/src/tailwind.css imports tokens.css + theme.css
```

### File roles

| File                       | Edit?                                    | Role                                                                                                  |
| -------------------------- | ---------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| `src/assets/tokens.json`   | **No** — Tokens Studio only              | Source token data                                                                                     |
| `src/tokens.generated.css` | **No** — regenerated by CI/build         | Style Dictionary CSS variables (`--button-primary-background-color-default-fill`, etc.)               |
| `src/tokens.theme.css`     | **No** — regenerated by `generate:theme` | Tailwind v4 `@theme inline` bridge                                                                    |
| `src/tokens.web.css`       | **Yes**                                  | Web platform extensions, focus ring, transitions, component aliases, values missing from Figma export |

### Package exports

| Import                                      | Resolves to                                           |
| ------------------------------------------- | ----------------------------------------------------- |
| `@ds-12/design-tokens/tokens.css`           | `src/tokens.web.css` (imports `tokens.generated.css`) |
| `@ds-12/design-tokens/tokens.generated.css` | `src/tokens.generated.css`                            |
| `@ds-12/design-tokens/theme.css`            | `src/tokens.theme.css`                                |

### Build commands

```bash
vp run @ds-12/design-tokens#build:tokens    # tokens.json → tokens.generated.css
vp run @ds-12/design-tokens#generate:theme  # build:tokens + tokens.theme.css
vp run @ds-12/design-tokens#build           # generate:theme + pack
```

### Component CSS token usage

- Prefer **generated** token names from `tokens.generated.css` in component CSS (e.g. `--button-primary-background-color-default-fill`, `--button-primary-type-small-padding-top`).
- Put **web-only** values (dimensions not in Tokens Studio export) in `tokens.web.css` (e.g. `--button-sm-height: 32px`).
- Do not recreate large alias layers in `tokens.web.css` when generated names can be referenced directly.
- Never hardcode colors or spacing in component CSS — always `var(--token-name)`.

## Component architecture

### File layout

```
packages/ui/src/components/<name>/
  index.tsx       # component, CVA, types
  <name>.css      # @utility blocks only
```

Register `<name>.css` in `packages/ui/src/tailwind.css` (do not side-effect-import CSS from `index.tsx`).

### CSS (`<name>.css`)

- `@utility <component>` — base layout (display, box-sizing, border-radius). No variant colors.
- `@utility <component>-<part>` — child slots (`badge-icon`, `button-label`).
- `@utility <component>-preset-…` or `@utility <component>-<variant>-<size>` — size/variant presets with token-backed properties.
- Variant colors and states live in preset utilities; use CSS nesting for `:hover`, `:disabled`, `[data-disabled]`, `:focus-visible`.
- No `@apply`. No hardcoded hex/px for design values (web-only exceptions go in `tokens.web.css`).

### CVA (`index.tsx`)

```ts
const fooVariants = cva("foo", {
  variants: { size: { sm: "", lg: "" }, variant: { a: "", b: "" } },
  compoundVariants: [{ size: "sm", variant: "a", class: "foo-preset-sm-a" }],
  defaultVariants: { size: "sm", variant: "a" },
});
```

- Prefer empty strings in `variants:` and real classes in `compoundVariants`.
- Some controls (e.g. `Input`) attach modifier utilities directly on boolean variants (`input-disabled`) — follow the existing component.

### DOM attributes

- `data-slot="<component>"` on root elements (required for tests and debugging).
- `data-variant={…}` where the component exposes a variant axis (not every component has it).

### Props

- Use discriminated unions when variant axes are mutually exclusive (see `Typography`).
- `className` merged last via `cn()` — import with a relative path: `import { cn } from '../../lib/utils.ts'`. Do not use the package alias `@ds-12/ui/utils`; no such export exists.

### Base UI

Interactive primitives wrap `@base-ui/react` (`Button`, `Avatar.Root`, `Field.Root`, etc.). DS-12 adds styling and thin composition only.

| ----------- | -------------- | ---------------------------- | ------------------------------------- |
| `display`   | forbidden      | forbidden                    | Fixed preset only                     |

## Naming conventions

### Token CSS variables

- Generated (Style Dictionary): `--<path-kebab>` e.g. `--button-primary-type-small-padding-top`, `--color-semantic-brand-bold`
- Web layer: `--<component>-<purpose>` e.g. `--button-sm-height`, `--badge-neutral-subtle-fill`

### `@utility` classes

| Pattern           | Example                                     |
| ----------------- | ------------------------------------------- |
| Base              | `badge`, `button`                           |
| Child part        | `badge-icon`, `button-label`                |
| Size preset       | `badge-preset-size-sm`                      |
| Variant preset    | `badge-neutral-subtle`, `button-primary-md` |
| Typography preset | `typography-preset-heading-h1-bold`         |

### Files

- Component folder: lowercase (`packages/ui/src/components/button/`)
- Co-located: `index.tsx` + `<name>.css`
- Stories: `apps/storybook/src/components/<name>/<name>.stories.tsx` or `apps/storybook/src/fields/<name>/…`
- Story fixtures: `<name>-story-fixtures.tsx` beside the story

## What NOT to do

### Tokens

- **Do not edit `tokens.json`** — controlled by Tokens Studio export/sync only.
- **Do not hand-edit `tokens.generated.css` or `tokens.theme.css`** — regenerated by `vp run @ds-12/design-tokens#generate:theme`.
- **Do not manually commit token CSS in feature PRs** — CI auto-commits drift as `chore: sync design tokens [skip ci]` when `tokens.json` changes.
- **Do not add large alias layers** in `tokens.web.css` when component CSS can reference generated token names directly.

### Components

- **Do not use `@apply`** anywhere in component CSS.
- **Do not hardcode** colors, spacing, or typography in component CSS.
- **Do not invent `compoundVariants`** — only combinations that exist in the component's CVA config and design tokens.
- **Do not pass invalid Typography props** (`size` on headings, `weight` on label/display) — blocked by discriminated unions; dev warnings in non-production.
- **Do not place stories in `packages/ui`** — stories live under `apps/storybook/src/`.
- **Do not duplicate component styling in apps** — compose `@ds-12/ui` components; import `@ds-12/ui/tailwind.css` for global styles.

### Imports

- Import tokens via `@ds-12/design-tokens/tokens.css` and `@ds-12/design-tokens/theme.css` — not from `@ds-12/ui`.

vp check && vp run -r test              # lint, format, typecheck, test all packages

### `vp run` vs `pnpm --filter` convention

- **Local / scripts / `package.json`:** use `vp run <pkg>#<task>` (e.g. `vp run @ds-12/design-tokens#generate:theme`). Vite+ handles caching and cross-package `dependsOn`.
- **GitHub Actions:** use `pnpm --filter <pkg> run <script>` for build/test commands (e.g. `pnpm --filter storybook run build-storybook`). Use `vp run <pkg>#<task>` only when invoking a Vite Task by name (e.g. `vp run @ds-12/design-tokens#generate:theme`).
- Do not mix the two for the same operation in the same context.
